What "fully dog-fooded" means

Design Intelligence Layer

  • essence.json — real v3 essence with archetype, theme DNA, guard mode
  • Blueprint — pages use real shells and layout zones
  • Theme — auradecantism style, resolves through theme registry
  • Guard rules — DNA and blueprint enforcement active
  • Drift detection — @decantr/vite-plugin watches for deviations

UI Framework Layer

  • @decantr/ui runtime — h(), mount(), signal reactivity
  • @decantr/ui components — sidebar, tabs, search all use real components
  • @decantr/css atoms — _flex, _col, _gap4, theme tokens
  • @decantr/ui router — hash-based routing for SPA
  • Compiler dev mode — hot-reload via compiler/dev.js

Why this matters

The workbench becomes a proof of the full Decantr stack. If the workbench works — essence → blueprint → shell → components → CSS atoms → guard rules — then the entire system works. Every bug you hit while building the workbench is a bug your users would have hit. Every rough edge you smooth is a rough edge removed for everyone.

It's also a reference implementation. When users ask "what does a real Decantr app look like?", you point them at the workbench source code.
