Delivery Phases

Phase 1: Foundation

Unblocks everything else
1a. packages/ui-catalog
  • Story schema + validation
  • Story renderer
  • Query API (getAll, getByCategory, search)
  • First 10-15 component stories (core components)
1b. apps/workbench — Isolation View
  • Real essence.json + full Decantr stack
  • Sidebar nav with category tree
  • Variant grid rendering from catalog
  • Theme/density/mode toolbar
  • Hot-reload via compiler dev mode

Phase 2: Depth

Rich developer experience
2a. Workbench — Playground View
  • Interactive prop editor from story controls
  • Live preview + code output panel
  • Signal state inspector
  • CSS atoms panel
2b. Remaining catalog stories
  • All 107 component stories
  • All 25 chart stories
  • Icon gallery story
  • CSS/theme stories

Phase 3: Polish & Public

Ship the showcase
3a. apps/ui-site — Showcase
  • Landing page + marketing pages
  • Component/chart/icon reference pages
  • Getting started guide
  • SSG build → GitHub Pages
  • ui.decantr.ai domain
3b. Workbench — Explorer View
  • Design system dashboard
  • Theme gallery (all themes side-by-side)
  • Density comparison
  • Accessibility audit (WCAG, colorblind)

Content Evolution

Building these apps will naturally reveal gaps in the existing content types. New archetypes, patterns, and shells will be created as needed — the content in decantr-content grows to match what the framework demands. This is the intended feedback loop:

Build real app → Discover missing content → Create archetype/pattern/shell → Publish to registry → Use in app

What stays the same

  • packages/ui, ui-chart, css — no moves, no restructuring. They're the source of truth.
  • apps/web — registry.decantr.ai stays as-is (Next.js, Supabase)
  • apps/api — Hono API stays as-is (Fly.io)
  • decantr-content — stays as separate repo, grows with new content types
  • All other packages — essence-spec, registry, core, mcp-server, cli, vite-plugin unchanged
