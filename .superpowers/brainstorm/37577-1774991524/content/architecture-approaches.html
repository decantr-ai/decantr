Architecture: How to organize the UI ecosystem in the monorepo

Three approaches for apps, workbench, and showcase — pick one or mix

A

Two Separate Apps

apps/ui-site/ — Public showcase site (decantr-native, static build → GitHub Pages)

apps/workbench/ — Local dev tool (component isolation, playground, explorer)

Clean separation of concerns. Showcase is optimized for public consumption (SEO, marketing, polished demos). Workbench is optimized for developer ergonomics (hot-reload, raw component rendering, debug tools). Different build pipelines, different audiences.

Pros

  • Each app is focused — no compromises
  • Workbench stays lean (no marketing/SEO overhead)
  • Showcase can be polished independently
  • Parallel development possible

Cons

  • Component demos written twice (workbench + showcase)
  • Two apps to maintain
  • Risk of drift between what workbench shows and what showcase shows
B

Unified App, Two Modes

apps/ui/ — Single Decantr-native app that runs in two modes:

• pnpm dev → workbench mode (local, full debug tools, hot-reload)

• pnpm build → showcase mode (static site for GitHub Pages)

Component demos are written once as "stories" (like Storybook). The workbench renders them with dev tools. The showcase renders them with marketing chrome. Same source, different presentation layers.

Pros

  • Write demos once, use everywhere
  • Zero drift between workbench and showcase
  • Single app to maintain
  • Shared component catalog format

Cons

  • More complex build configuration
  • Showcase and workbench needs may diverge over time
  • Dev mode carries showcase code weight (minor)
C

Shared Catalog + Two Thin Shells

packages/ui-catalog/ — New package: component stories, metadata, demo definitions

apps/workbench/ — Thin shell: renders catalog with dev tools

apps/ui-site/ — Thin shell: renders catalog with marketing/docs chrome

The "write once" benefit of B, but with clean separation of A. The catalog package is the single source of truth for all component demos. Each app is a thin rendering shell that imports the catalog and adds its own chrome.

Pros

  • Write demos once (in the catalog package)
  • Each app stays focused and lean
  • Catalog is independently testable/publishable
  • Could later power a third consumer (e.g., Figma plugin, docs generator)

Cons

  • Three things to maintain instead of one or two
  • Catalog package needs its own schema/conventions
  • Slight indirection (apps import catalog → catalog imports ui)

★ Recommendation: Option C

Option C gives you the best of both worlds. The catalog package becomes your "component story" format — each component gets a demo file that describes its variants, props, and usage examples. This is the same pattern that made Storybook successful, but native to your framework.

The workbench stays laser-focused on developer ergonomics (hot-reload, debug panels, theme switching). The showcase site stays focused on public presentation (marketing copy, SEO, polished demos). Neither app carries the other's baggage.

And critically: the catalog package could later be published to npm, letting other developers contribute demos or use the catalog format for their own component libraries built on @decantr/ui.
