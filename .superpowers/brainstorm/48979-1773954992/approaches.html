Implementation Approaches

How should the generator know which decorators to apply?

A

Pattern Type Declaration

Patterns declare their effect_type in JSON. Generator maps types to decorators via recipe config.

Pros

  • Explicit, predictable behavior
  • Patterns opt-in to effects
  • Easy to extend with new types
  • Recipe controls mapping

Cons

  • Requires updating pattern JSONs
  • New field to maintain
// pattern.json
{
  "id": "bento-features",
  "effect_types": ["code_preview", "stat_display"]
}

// recipe.json
"visual_effects": {
  "type_mapping": {
    "code_preview": "d-terminal-chrome",
    "stat_display": "d-glow-primary"
  }
}
B

Component Detection

Generator detects which components a pattern uses (pre, Statistic, Card) and applies decorators based on component type.

Pros

  • No pattern changes needed
  • Works with existing patterns
  • Automatic for new patterns

Cons

  • Less explicit control
  • May apply effects unexpectedly
  • Harder to opt-out
// recipe.json
"visual_effects": {
  "component_mapping": {
    "pre": "d-terminal-chrome",
    "Statistic": "d-glow-primary",
    "Card": "d-glass"
  }
}

// Generator detects: pattern uses <pre>
// Generator applies: d-terminal-chrome
C

Hybrid: Types + Component Fallback

Patterns CAN declare effect_types (explicit). If not declared, generator falls back to component detection (implicit).

Pros

  • Best of both worlds
  • Existing patterns work immediately
  • New patterns can be explicit
  • Graceful migration path

Cons

  • Two code paths to maintain
  • Slightly more complex logic
// Generator logic:
1. Check pattern.effect_types? Use explicit mapping
2. Else detect components in pattern code
3. Apply decorators via recipe config

// Allows gradual adoption:
// - Old patterns: auto-detection
// - New patterns: explicit types

My recommendation: C (Hybrid) — Works immediately with existing patterns via component detection, but allows explicit control for new patterns. Best migration path.
