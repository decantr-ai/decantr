## 1. Context and Motivation

The decantr-content registry now contains 29 blueprints covering AI agents, CRM, observability, healthcare, marketplaces, events, data engineering, and more. Only 3 have showcase implementations (agent-marketplace, registry-platform, terminal-dashboard). The remaining 26 have no visual demonstration.

Showcases serve as **marketing demos** — visitors browse them on decantr.ai to see what each blueprint produces. Visual polish is everything. Mock data, no real functionality. Every route defined in the blueprint must be built.

Each showcase is scaffolded via the canonical `decantr init --blueprint=X` pipeline, proving the composition system works end-to-end: blueprint → archetypes → patterns → themes → section contexts → AI-generated React code.

## 2. Architecture

Each showcase is a **fully standalone Vite + React 19 SPA** with zero cross-project dependencies (no shared chrome, no shared packages beyond `@decantr/css`).

### Key Principles

- **Blueprint-verbatim**: Every route in the blueprint gets a page. No shortcuts, no "hero screens only."
- **CLI-first**: `decantr init` generates the essence, CSS, DECANTR.md, and section contexts. The AI reads those files to build React code — the canonical flow.
- **Standalone**: Each project is independently buildable, deployable, and browsable. No inter-showcase dependencies.
- **Zone-based routing**: Routes are grouped by topology zone (public → gateway → primary → auxiliary), each with its own shell layout.
- **Mock everything**: Auth via localStorage flag. Data from typed mock objects. No API calls, no backend.
- **Static hosting**: HashRouter for compatibility with `/showcase/{name}/` subpath on the registry site.

### Stack (identical per project)

| Dependency | Version | Purpose |
|-----------|---------|---------|
| `react` | ^19.0.0 | UI library |
| `react-dom` | ^19.0.0 | DOM rendering |
| `react-router-dom` | ^7.0.0 | HashRouter, Routes, Link |
| `@decantr/css` | workspace:* | CSS atoms runtime |
| `lucide-react` | ^0.400.0 | Icon library (matches personality "Lucide icons" in all blueprints) |
| `vite` | ^6.0.0 | Build tool |
| `@vitejs/plugin-react` | ^4.0.0 | React Fast Refresh |
| `typescript` | ^5.7.0 | Type checking |

No other dependencies. No UI frameworks (no shadcn, no Radix). Components are hand-built from the pattern specs in section context files, styled with `@decantr/css` atoms + theme tokens + treatment/decorator classes. Note: `lucide-react` is included for all blueprints whose personality includes "Lucide icons" (nearly all). Terminal-themed projects may omit it if the personality calls for ASCII-only iconography.

## 4. Build Process Per Showcase

Each showcase follows this exact sequence:

### Step 1: Scaffold via CLI

```bash
cd apps/showcase/
mkdir {blueprint-id}
cd {blueprint-id}
decantr init --blueprint={blueprint-id} --yes
```

This generates:
- `decantr.essence.json` (V3.1 with sections, routes, DNA, personality)
- `DECANTR.md` (design guide with guard rules, CSS approach, atom reference)
- `.decantr/context/scaffold.md` (topology, voice, route map, shared components)
- `.decantr/context/section-{id}.md` (per-section: shell, patterns, decorators, spacing)
- `src/styles/tokens.css` (theme custom properties)
- `src/styles/treatments.css` (base treatments + personality utilities)
- `src/styles/global.css` (reset + layer declarations)

### Step 2: Create project boilerplate

Create `package.json`, `vite.config.ts`, `tsconfig.json`, `index.html`, `src/main.tsx` from templates above.

### Step 3: Build shells

Read each section context file. Extract the shell spec (regions, internal_layout, dimensions, responsive behavior). Build a React shell component for each unique shell.

Shells used across the registry:

| Shell | Used By (Blueprint Count) |
|-------|--------------------------|
| `sidebar-main` | 22 blueprints |
| `top-nav-footer` | 25 (marketing sections) |
| `centered` | 24 (auth sections) |
| `top-nav-main` | 8 |
| `minimal-header` | 5 |
| `full-bleed` | 3 |
| `sidebar-aside` | 4 |
| `terminal-split` | 2 |
| `chat-portal` | 1 (carbon-ai-portal) |
| `copilot-overlay` | 1 (ai-copilot-shell) |
| `three-column-browser` | 2 (knowledge-base, api-reference) |
| `sidebar-right` | 1 |
| `top-nav-sidebar` | 1 |
| `workspace-aside` | 1 (realtime-collaboration) |
| `sidebar-main-footer` | 1 |

Each showcase builds its own shell implementations. Even though `sidebar-main` appears in 22 blueprints, each implementation is themed differently (different decorators, tokens, spatial values). The section context files provide the exact specs per blueprint.

### Step 4: Build page components

For each route in the blueprint, create a page component that:
1. Reads the patterns assigned to that page (from the section context)
2. Implements each pattern using the visual_brief, composition DSL, slots, and components
3. Uses mock data from `data/mock.ts`
4. Applies the correct treatment and decorator classes
5. Follows the responsive breakpoints from the pattern spec

### Step 5: Build shared components

Patterns that appear on 2+ pages within the same showcase are extracted as shared components in `src/components/`. The scaffold.md file lists these under "Shared Components."

### Step 6: Build mock data

Create `src/data/mock.ts` with typed interfaces and realistic mock data for every page. Data should feel authentic:
- Use realistic names, dates, amounts
- Include various states (active, pending, error, completed)
- Provide enough items for lists/tables (8-15 rows typical)
- Match the blueprint's domain (medical terms for health, financial data for fintech, etc.)

### Step 7: Wire routing

Create `App.tsx` with:
- `AuthContext` provider
- `HashRouter` with all routes grouped by zone
- Each zone wrapped in its shell component
- Auth-protected routes redirect to `/login` when unauthenticated
- Public routes accessible without auth

### Step 8: Generate decorator CSS

The `treatments.css` from `decantr init` includes base treatments and personality utilities. The decorator CSS (theme-specific classes like `health-card`, `fin-metric`, `term-panel`) is AI-generated from the structured `decorator_definitions` in the theme. Write these to `src/styles/decorators.css` wrapped in `@layer decorators`.

### Step 9: Build and commit

```bash
pnpm install
pnpm build    # tsc && vite build
```

Commit the entire project including `dist/`.

## 7. Shell Implementations

### Shell Implementation Approach

Each shell is a React layout component that:
1. Renders the regions defined in the shell spec (nav, header, body, footer, aside, etc.)
2. Uses CSS Grid or Flexbox per the shell's `layout` type
3. Applies the shell's `internal_layout` spatial specs (width, height, padding, gap, border)
4. Implements responsive behavior (collapse sidebar, hide aside, etc.)
5. Passes children via an `<Outlet />` or slot props pattern

### Shell Template Pattern

```tsx
// Example: SidebarMain shell
export function SidebarMain({ children }: { children: React.ReactNode }) {
  const [navExpanded, setNavExpanded] = useState(true);

  return (
    <div className="_grid _h[100vh]" style={{
      gridTemplateColumns: navExpanded ? '240px 1fr' : '64px 1fr',
      gridTemplateRows: '52px 1fr',
      gridTemplateAreas: '"nav header" "nav body"',
    }}>
      <header style={{ gridArea: 'header' }} className="d-surface _flex _items-center _px4 _border-b">
        {/* Header content */}
      </header>
      <nav style={{ gridArea: 'nav' }} className="d-surface _flex _col _border-r _overflow-y-auto">
        {/* Navigation items */}
      </nav>
      <main style={{ gridArea: 'body' }} className="_overflow-y-auto _p6">
        {children}
      </main>
    </div>
  );
}
```

The exact dimensions, decorators, and responsive breakpoints come from each project's section context files. The shell spec provides the structural template; the theme provides the visual treatment.

## 9. Visual Quality Standards

### Every page must meet these criteria:

1. **No layout shift on load** — All content areas have defined heights/widths
2. **Realistic content density** — Tables have data, grids have cards, charts have lines
3. **Consistent spacing** — Uses the spacing scale from the theme's spatial config
4. **Theme coherence** — Every surface, card, badge, and button uses the correct decorator class
5. **Typography hierarchy** — Headings, body, muted text, monospace data all follow the theme's type scale
6. **Status semantics** — Green/amber/red (or theme equivalents) used consistently for status indicators
7. **Empty states** — Not needed (mock data fills everything), but skeleton loaders can show on initial "load"
8. **Hover/focus states** — Interactive elements have visible hover states using treatment classes
9. **Icon consistency** — Lucide icons throughout, matching the "Lucide icons" personality directive
10. **Dark/light correctness** — Colors work in the specified mode, no contrast issues

### Visual Polish Checklist (per page)

- [ ] All cards use the correct theme decorator (e.g., `health-card`, `fin-card`, `term-panel`)
- [ ] All text uses semantic color tokens (`--d-text`, `--d-text-muted`, `--d-primary`)
- [ ] All surfaces use layered backgrounds (`--d-bg`, `--d-surface`, `--d-surface-raised`)
- [ ] All interactive elements have `d-interactive` treatment
- [ ] All data displays use `d-data`, `d-header`, `d-row`, `d-cell` treatments
- [ ] Spacing follows the density scale from the section context
- [ ] Border radius matches the theme's radius philosophy
- [ ] Motion/animation matches the theme's motion preference

## 10. Routing and Navigation

### Zone-Based Route Organization

Every blueprint's routes are organized by topology zone:

```tsx
<HashRouter>
  <Routes>
    {/* PUBLIC zone — marketing, about, legal */}
    <Route element={<PublicShell />}>
      <Route path="/" element={<Home />} />
      <Route path="/about" element={<About />} />
    </Route>

    {/* GATEWAY zone — auth */}
    <Route element={<CenteredShell />}>
      <Route path="/login" element={<Login />} />
      <Route path="/register" element={<Register />} />
      {/* ... other auth routes */}
    </Route>

    {/* PRIMARY + AUXILIARY zones — app (auth required) */}
    <Route element={<AuthGuard><AppShell /></AuthGuard>}>
      <Route path="/dashboard" element={<Dashboard />} />
      <Route path="/settings/*" element={<Settings />} />
      {/* ... all app routes */}
    </Route>
  </Routes>
</HashRouter>
```

### Navigation Items

Sidebar/nav items are derived from the blueprint's routes. Each shell renders navigation links that match the routes for its zone. Active route highlighting uses `useLocation()` from react-router-dom.

### Route-to-Page Mapping

The blueprint's `routes` object is the source of truth:

```json
{
  "/dashboard": { "page": "overview", "shell": "sidebar-main", "archetype": "tenant-switcher-dashboard" },
  "/members": { "page": "members", "shell": "sidebar-main", "archetype": "tenant-switcher-dashboard" }
}
```

Every route gets a page component. The `shell` determines which layout wrapper. The `archetype` determines which section context to read for pattern specs.

## 11. Auth Simulation

### Mock Auth Pattern

```tsx
// AuthContext in App.tsx
const AuthContext = createContext<{
  isAuthenticated: boolean;
  login: () => void;
  logout: () => void;
}>({ isAuthenticated: false, login: () => {}, logout: () => {} });

function AuthProvider({ children }: { children: React.ReactNode }) {
  const [isAuthenticated, setIsAuthenticated] = useState(
    () => localStorage.getItem('decantr_authenticated') === 'true'
  );

  const login = () => {
    localStorage.setItem('decantr_authenticated', 'true');
    setIsAuthenticated(true);
  };

  const logout = () => {
    localStorage.removeItem('decantr_authenticated');
    setIsAuthenticated(false);
  };

  return (
    <AuthContext.Provider value={{ isAuthenticated, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
}
```

### Auth Form Behavior

- Login form: any email/password → calls `login()`, redirects to primary zone
- Register form: any data → calls `login()`, redirects to primary zone
- MFA screens: show the UI, any code works
- Logout: calls `logout()`, redirects to `/login`

### Route Protection

```tsx
function AuthGuard({ children }: { children: React.ReactNode }) {
  const { isAuthenticated } = useAuth();
  const location = useLocation();

  if (!isAuthenticated) {
    return <Navigate to="/login" state={{ from: location }} replace />;
  }

  return children;
}
```

## 12. Theme and CSS Integration

### CSS Layer Cascade

Every showcase uses the same layer order declared in `global.css`:

```css
@layer reset, tokens, treatments, decorators, utilities, app;
```

### File Responsibilities

| File | Layer | Source | Contents |
|------|-------|--------|----------|
| `global.css` | reset | Generated by CLI | Box-sizing, body, focus-visible, sr-only |
| `tokens.css` | tokens | Generated by CLI | CSS custom properties for colors, spacing, radii, shadows |
| `treatments.css` | treatments + utilities | Generated by CLI | Base treatment classes (d-interactive, d-surface, etc.) + personality utilities |
| `decorators.css` | decorators | AI-generated from theme's decorator_definitions | Theme-specific classes (health-card, fin-metric, term-panel, etc.) |

### Decorator Generation

The decorator CSS is NOT generated by the CLI. It's generated by the AI during page building, reading the `decorator_definitions` from the section context files. Each definition provides:
- `intent`: When to use the decorator
- `suggested_properties`: CSS property/value pairs
- `pairs_with`: Other decorators that complement it
- `usage`: Contexts where it applies

The AI writes the decorator classes to `src/styles/decorators.css` wrapped in `@layer decorators`.

### Theme-Specific Considerations

| Theme | Special Handling |
|-------|-----------------|
| `terminal` | All text is monospace. ASCII borders. Scanline effects optional. Green-on-black. |
| `healthcare` | WCAG AAA contrast. No information by color alone. Large touch targets. |
| `government` | 7:1 contrast ratio. 3px focus outlines. 16px minimum body text. Minimal motion. |
| `fintech` | Monospace numbers. Gain/loss semantic colors. Dense layouts. Instant transitions. |
| `neo-tokyo` | Gradient borders. Pill shapes. Bouncy spring animations. Bold 800-weight headings. |
| `dopamine` | Saturated gradients. Y2K maximalism. Bouncy motion. High energy. |
| `earth` | Grain texture overlay. Organic SVG dividers. Serif headings. Slow transitions. |
| `glassmorphism` | Backdrop-blur surfaces. Frosted glass cards. Atmospheric gradients. |
| `editorial` | Zero decoration. Dramatic type scale. Flat surfaces. Instant transitions. |
| `paper` | Warm whites. Comfortable reading measure. Presence colors for collaboration. |

## 13. Responsive Behavior

### Breakpoints (from @decantr/css)

| Breakpoint | Width | Common Changes |
|-----------|-------|----------------|
| `sm` | 640px | Mobile → tablet transition |
| `md` | 768px | Sidebar collapse threshold |
| `lg` | 1024px | Three-column layouts activate |
| `xl` | 1280px | Aside panels appear |

### Responsive Patterns

- **Sidebar shells**: Collapse to 64px rail at `md`, hide entirely below `sm`
- **Three-column browser**: Show all three at `lg`, collapse nav at `md`, single column below `sm`
- **Copilot overlay**: Side panel at `lg`, bottom sheet below `lg`
- **Data tables**: Horizontal scroll on mobile
- **Card grids**: 3 columns → 2 columns → 1 column
- **Charts**: Maintain aspect ratio, reduce padding

Responsive behavior comes from each pattern's `responsive` spec in the section context.

### Per-Batch Workflow

1. **Pre-scaffold**: Run `decantr init` for each project in the batch (can be done in parallel)
2. **Parallel agent builds**: Launch one agent per project to build all React code from the generated context files
3. **Validate**: Each agent runs `pnpm build` to confirm TypeScript compiles and Vite builds
4. **Commit**: Batch commit all projects in the batch

### Agent Prompt Structure

Each build agent receives:
- The blueprint ID and theme
- The generated `scaffold.md` content (topology, voice, routes)
- All section context files for the blueprint
- The decorator_definitions from the theme
- The exact route list from the blueprint JSON
- The mock data requirements for the domain

The agent produces:
- All source files in `src/`
- Decorator CSS in `src/styles/decorators.css`
- A working `dist/` from `vite build`

## 15. Infrastructure and Build Pipeline

### Monorepo Integration

Each showcase is a workspace member via `pnpm-workspace.yaml`'s `apps/showcase/*` glob. No changes needed to the workspace config — new directories are auto-discovered.

### Build Commands

Existing commands handle everything:

```bash
# Build all showcases (including new ones)
pnpm showcase:build

# Copy all dist/ to registry app
pnpm showcase:copy

# Both
pnpm showcase
```

The `copy-showcase-dist.mjs` script auto-discovers all directories in `apps/showcase/` that have a `dist/` folder. No script modifications needed.

### Deployment

1. `pnpm showcase` builds and copies all 29 showcases
2. Registry app (decantr.ai) build includes the static showcase files
3. Deploy to Vercel serves them at `/showcase/{blueprint-id}/`
