# Blueprint Showcase System Design

**Date:** 2026-03-31
**Status:** Draft
**Author:** David Aimi

---

## Overview

A live demo showcase system within the Decantr registry web portal (`apps/web`) that lets end users view fully scaffolded, interactive demos of blueprint compositions. Each showcase is a real Decantr-initted project that has gone through the full design pipeline — proving blueprints work end-to-end.

### Goals

- Interactive demos of blueprints with mock data (client-side interactivity, simulated responses)
- Each showcase is a standalone project scaffolded via `decantr init` from a blueprint
- Served from the existing Vercel deployment ($0 incremental hosting cost)
- Showcases validate the full Decantr pipeline: `decantr init` → essence → AI code generation → guard
- Maintainable within the existing monorepo CI

### Non-Goals

- Fully functional backends (auth, real databases, real API calls)
- Separate Vercel projects or subdomains per showcase
- Auto-generation of showcase code without human/AI review
- Supporting non-React frameworks for showcases (React only for now)

## Architecture

### Monorepo Structure

Showcase projects live as standalone apps under `apps/showcase/`, each with their own `package.json`:

```
decantr-monorepo/
  pnpm-workspace.yaml          ← updated: add apps/showcase/*
  apps/
    web/                        ← existing Next.js portal
      public/
        showcase/
          carbon-ai-portal/     ← built static output (copied from showcase dist/)
          terminal-dashboard/
    api/                        ← existing Hono API
    showcase/
      carbon-ai-portal/         ← standalone Decantr project
        package.json
        vite.config.ts
        index.html
        decantr.essence.json    ← generated by decantr init
        DECANTR.md              ← generated by decantr init
        .decantr/               ← generated by decantr init
          project.json
          context/
            task-scaffold.md
            task-add-page.md
            task-modify.md
            essence-summary.md
          cache/
          custom/
        src/
          styles/
            tokens.css          ← generated by decantr init (theme CSS custom properties)
            decorators.css      ← generated by decantr init (recipe CSS classes)
          pages/                ← AI-generated React pages following essence spec
          components/           ← AI-generated React components following pattern specs
          mock-data/            ← domain-appropriate typed mock data
          main.tsx              ← Vite entry point
          App.tsx               ← Router + shell layout
      terminal-dashboard/       ← another standalone project
        package.json
        vite.config.ts
        ...
```

### Why Standalone Projects

Each showcase is its own project (not Next.js route groups) because:

1. **Pipeline validation** — Each project has its own `decantr.essence.json`, `DECANTR.md`, and `.decantr/` directory, proving `decantr init` works for that blueprint
2. **Duality** — The showcase serves as both a demo for end users AND a scaffolding validation artifact for the developer
3. **Isolation** — Each has its own dependency tree, build, and guard validation
4. **Authenticity** — The project structure mirrors what an end user would create with `decantr init`

### Technology Stack Per Showcase

| Layer | Technology | Rationale |
|-------|-----------|-----------|
| Build | Vite | `@decantr/vite-plugin` exists for drift detection; lightweight static output |
| UI | React | Web portal is React-based; most common target framework |
| Styling | `@decantr/css` atoms + `tokens.css` + `decorators.css` | Framework-agnostic CSS from the Decantr system |
| Routing | React Router (client-side) | Blueprints define routes; client-side routing for static demos |
| Mock data | TypeScript modules | Typed mock data per domain, no server required |
| Drift detection | `@decantr/vite-plugin` | Validates code stays aligned with essence spec during dev |

### Serving Strategy

Built showcase output (static HTML/CSS/JS from `vite build`) is copied to `apps/web/public/showcase/{blueprint-name}/`. Vercel serves these as static files from the same deployment at `/showcase/{blueprint-name}/index.html`.

The web app's registry detail page for blueprints adds a "View Demo" external link icon pointing to `/showcase/{blueprint-name}/`.

### Showcase Chrome

Each showcase includes its own minimal header (baked into the showcase app itself, not the web app's layout):

- **Left:** "decantr.ai" text linking back to `/registry`
- **Right:** Dropdown to switch between available showcases (navigates to `/showcase/{other-blueprint}/`)
- **Below header:** Full-viewport demo content with the blueprint's own shell/navigation

The showcase chrome is a shared React component across all showcases, imported from a shared package or copied into each project.

## Lifecycle: Creating a New Showcase

### Step 1 — Create the project

```bash
mkdir -p apps/showcase/carbon-ai-portal
cd apps/showcase/carbon-ai-portal
npm init -y
# Install deps: react, react-dom, react-router-dom, vite, @vitejs/plugin-react
# Install decantr deps: @decantr/css, @decantr/vite-plugin
```

### Step 2 — Initialize with Decantr

```bash
decantr init
# Select blueprint: carbon-ai-portal
# CLI resolves: blueprint → 7 archetypes → patterns → carbon theme → carbon recipe → shells
# CLI outputs: decantr.essence.json, DECANTR.md, .decantr/, tokens.css, decorators.css
```

### Step 3 — AI generates code

An AI assistant reads the generated `DECANTR.md` and `decantr.essence.json`, then generates:

- React page components matching the blueprint's route map
- Shell layouts matching the shell specs (sidebar-main, chat-portal, top-nav-main, etc.)
- Pattern compositions matching the archetype page definitions
- Components using `@decantr/css` atoms and `tokens.css` custom properties
- Mock data appropriate to the blueprint's domain

### Step 4 — Validate

```bash
decantr check
# Guard validates generated code against essence spec
# DNA rules: style, recipe, density, accessibility, theme-mode
# Blueprint rules: structure, layout, pattern existence
```

### Step 5 — Build and integrate

```bash
pnpm build                  # vite build → dist/
# CI copies dist/ to apps/web/public/showcase/carbon-ai-portal/
```

### Step 6 — Commit

The entire project (source + decantr files + built output) is committed to the monorepo.

## Build Automation

### Workspace Configuration

Update `pnpm-workspace.yaml`:

```yaml
packages:
  - 'packages/*'
  - 'apps/*'
  - 'apps/showcase/*'
```

### Root Scripts

```json
{
  "scripts": {
    "showcase:build": "pnpm --filter './apps/showcase/*' build",
    "showcase:deploy": "node scripts/copy-showcase-dist.js"
  }
}
```

### `scripts/copy-showcase-dist.js`

A simple Node.js script that:

1. Reads all directories under `apps/showcase/`
2. For each, copies `dist/` contents to `apps/web/public/showcase/{name}/`
3. Runs before the web app build in CI

### CI Integration

Addition to `.github/workflows/ci.yml`:

```yaml
- name: Build showcases
  run: pnpm showcase:build

- name: Copy showcase dist to web public
  run: node scripts/copy-showcase-dist.js

- name: Build web
  run: pnpm --filter decantr-web build
```

The existing CI already runs on push/PR to main. Showcase builds are added before the web build so static files are present when the web app builds.

### Guard Validation in CI

Optional addition — run `decantr check` on each showcase in CI to catch drift:

```yaml
- name: Validate showcases
  run: |
    for dir in apps/showcase/*/; do
      (cd "$dir" && npx decantr check) || exit 1
    done
```

## Web App Integration

### Registry Detail Page

The registry detail page for blueprints (`/registry/blueprints/@official/{slug}`) gains a "View Demo" button when a showcase exists for that blueprint:

```tsx
{hasShowcase && (
  <a href={`/showcase/${slug}/`} target="_blank" rel="noopener">
    <ExternalLinkIcon /> View Demo
  </a>
)}
```

The list of available showcases is derived from a static config file or by checking which directories exist in `public/showcase/`.

### Showcase Discovery

A new section on the registry page (or a dedicated `/showcase` listing page) displays all available showcases as cards with:

- Blueprint name and description
- Theme preview (color swatch from theme seed)
- "View Demo" link
- Blueprint metadata (archetype count, route count)

| Aspect | Validation |
|--------|-----------|
| Blueprint composition | Routes, archetype merging, pattern references resolve correctly |
| Theme generation | `tokens.css` produces valid, usable CSS custom properties |
| Recipe decoration | `decorators.css` produces valid CSS classes matching recipe spec |
| Shell layouts | Generated pages faithfully implement shell specs |
| Pattern fidelity | Components follow pattern definitions (components, presets, layout) |
| Guard rules | `decantr check` passes on the generated code |
| CLI scaffolding | `decantr init` correctly resolves and scaffolds each blueprint |
| Full pipeline | init → essence → AI code → guard → working app → deployed demo |

## Maintenance

### When blueprints change

If a blueprint spec changes in `decantr-content`:

1. Re-run `decantr init` in the affected showcase (or `decantr upgrade` when `--apply` is implemented)
2. AI re-generates affected pages/components
3. Run `decantr check` to validate
4. Rebuild and commit

### When patterns/themes/recipes change

If a pattern, theme, or recipe changes:

1. Run `decantr sync` in the showcase to update the local cache
2. Run `decantr check` to detect drift
3. Fix any violations
4. Rebuild and commit

### Shared showcase chrome

The showcase header (decantr.ai branding + blueprint switcher dropdown) is a shared component. Options:

- **Copy into each project** — simplest, slight duplication
- **Shared workspace package** (`packages/showcase-chrome`) — DRY, but adds a dependency

Start with copy-paste for the first 1-2 showcases. Extract to a shared package once patterns stabilize.

## Open Questions

1. **Should built `dist/` output be committed to the repo or generated in CI only?** Committing makes local dev easier but bloats the repo. CI-only keeps the repo clean but requires a build step to preview showcases.

2. **Should the showcase chrome (header + switcher) live as a shared package or be copied per project?** Start with copy, extract when stable.

3. **Should `decantr check` run in CI as a hard gate or advisory?** Start advisory (warnings), promote to gate once showcases are stable.
